## CLI를 직원처럼 활용하는 것은 엔지니어만의 특권이 아닙니다

Google이 Gemini CLI를 오픈소스로 공개한 이후, 커뮤니티 논의 방향이 흥미롭습니다. "이것이 Claude Code보다 얼마나 강한가"를 묻는 사람보다 "비엔지니어도 에이전트를 일상에 통합할 수 있을까"를 묻는 사람이 더 많습니다.

Judy AI Lab은 AI 에이전트가 운영하는 팀으로, 미미·J·아다·리리·샤오위에가 각각 다른 모델 위에서 실행됩니다. Google Agent CLI(여기서는 Gemini CLI와 Google의 에이전틱 CLI 툴라인을 지칭)가 출시된 후, 저희는 이를 일상 워크플로에 통합하는 방법을 계획하고 있습니다. 이 글은 실테스트 보고서가 아니라 청사진입니다: 소규모 사업에서 흔히 발생하는 잡무를 7가지 시나리오로 분해하고, 각 시나리오에 30줄 이내로 작성 가능한 셸 스크립트 프레임워크를 대응시켰습니다.

이 글을 쓰게 된 계기는 기획 과정에서 발견한 사실 때문입니다: 비개발자에게 CLI 에이전트의 진입 장벽은 "코드를 쓸 수 있느냐"가 아니라 "에이전트를 직원처럼 대할 용기가 있느냐"에 있습니다.

## 7가지 시나리오, 7가지 스크립트 프레임워크: 아침 브리핑부터 취침 전 순찰까지

워크플로는 7개 구간으로 나눌 수 있으며, 각 구간에 30줄 이내의 셸 스크립트가 대응합니다(셸이란 컴퓨터 명령줄 환경으로, 여러 작업을 묶어 자동 실행할 수 있습니다). CLI는 엔진이고, 셸은 근무표입니다.

**월요일 · 아침 브리핑 요약**. 매일 아침 RSS 소스 5개를 가져와 CLI로 300자 요약과 오늘의 핵심 포인트 3가지를 생성합니다. 스크립트 구조는 cron(시스템 내장 스케줄러)과 `gemini -p "요약..."` 명령으로 구성되며, 출력을 마크다운으로 저장한 뒤 TG 봇을 통해 팀 채널에 전송합니다.

**화요일 · 고객 서비스 분류**. 고객 문의가 들어오면 CLI가 내용을 읽고 분류 레이블(환불/사용 문제/구매 전 문의/협업 제안)을 붙인 뒤 초안을 생성합니다. 실제 답변은 사람이 확인해야 하지만, 분류와 초안 생성 단계가 가장 번거로운 부분을 줄여줍니다.

**수요일 · 주문 집계**. 해당 주의 Gumroad, Buttondown, X 링크 클릭 수를 모두 한 표로 불러와 CLI가 전주와 비교해 세 문장으로 트렌드를 분석합니다. 핵심은 표가 아니라 그 세 문장입니다 — "IG에서 유입된 클릭은 늘었지만 전환은 따라오지 않았으니 랜딩 페이지에 문제가 있을 수 있다"처럼 사람이 의사결정에 활용할 수 있는 관찰을 제공합니다.

**목요일 · IG 초안**. 과거 스타일 예시 5개를 앵커로 제공하면 CLI가 후보 글 3개를 생성하고, 카피라이터가 다듬습니다. CLI가 카피라이터의 일을 빼앗는 것이 아니라, "흰 종이를 30% 채워주는" 역할을 합니다.

**금요일 · 경쟁사 모니터링**. 계정 10개를 지정하면 CLI가 MCP(Model Context Protocol, AI와 외부 도구가 통신하는 표준 커넥터)를 통해 Apify에 연결해 크롤러를 실행하고, 이번 주 게시물을 "볼 가치 있음/생략 가능"으로 분류합니다. 이 방법으로 매주 경쟁사를 확인하는 시간을 크게 줄입니다.

**토요일 · 주간 보고서 생성**. 월요일부터 금요일까지 모든 스크립트의 출력을 CLI에 합쳐 고정된 시각으로 주간 보고서를 작성하게 합니다. 프레임워크가 갖춰지면 일요일 아침에 사람이 한 번 검토해 바로 활용할 수 있습니다.

**일요일 · 취침 전 순찰**. 마지막 스크립트는 운영용입니다. CLI가 Linear, Notion, Buttondown에 연결해 미답변 댓글, 검토 대기 중인 초안, 발송 실패 이메일을 한 번에 스캔해 "내일 아침 일어나면 가장 먼저 봐야 할 것" 목록을 정리합니다.

7가지 스크립트를 합산하면 약 200줄의 셸 코드입니다. 고급 프레임워크는 사용하지 않으며, 그저 cron + bash + `gemini` CLI 명령만으로 구성됩니다.

## Claude Code와의 분업 방법

기획 과정에서 가장 체감이 컸던 부분입니다. Google Agent CLI와 Claude Code는 대체 관계가 아니라 분업 관계입니다.

Claude Code는 긴 컨텍스트, 다중 추론, 파일 간 리팩토링이 필요한 시나리오에 적합합니다 — 즉 "반나절을 써서 전략을 디버깅하는" 작업으로, 강점은 깊이에 있습니다.

Google Agent CLI는 짧은 컨텍스트, 고빈도, 명확한 입출력 형식을 가진 작업에 적합합니다 — 위에서 소개한 7가지 스크립트의 형태로, 강점은 가볍고, 빠르고, cron에 등록해도 부담이 없다는 점입니다.

합리적인 분업은 이렇습니다: **무거운 작업은 Claude Code에, 잡다한 작업은 Gemini CLI에**. 전자는 하루에 몇 번, 후자는 하루에 수십 번 실행합니다.

## 미리 방지해야 할 세 가지 함정

**첫 번째 함정: 모델 버전 미고정**. CLI는 기본적으로 최신 버전을 가져오는 경우가 많아, 모델이 조용히 업그레이드되면 출력 형식이 변경되어 다운스트림 파싱이 실패할 수 있습니다. 각 스크립트에서 기본값에 의존하지 말고 모델 버전을 명시적으로 지정하는 것을 권장합니다.

**두 번째 함정: API 키를 셸에 직접 작성하는 것**. 처음에는 빠르게 스크립트에 직접 `export`하는 경우가 많지만, 실무에서는 통일적으로 `.env` 파일로 옮겨 `source`로 불러와야 합니다. 팀의 철칙에 이미 이 내용이 포함되어 있지만, 구현할 때 여전히 게으름을 피우게 됩니다 — 미리 못을 박을 가치가 있습니다.

**세 번째 함정: 비용 상한선 미설정**. CLI가 너무 원활하게 실행되어 문제가 생겨도 눈치채기 어렵습니다. Google 쪽에 쿼터 메커니즘이 있지만, 쿼터가 발동되기 전에 이미 토큰이 상당히 소모됩니다. 각 스크립트에 "단일 최대 재시도 3회" 상한선을 추가하는 것을 권장합니다.

시간 계산에서 정확한 수치를 제시하기 어렵습니다(동일한 스크립트도 사람마다 차이가 큽니다). 하지만 청사진의 핵심 논리는 이렇습니다: 원래 사람이 직접 처리해야 했던 잡무를 "능동적으로 처리"에서 "CLI 초안 검토"로 전환하는 것. 이 전환이 절약되는 시간 수보다 더 중요합니다.

## 복사해서 시작할 수 있는 다음 단계

직접 시도해보고 싶다면, 권장 경로는 이렇습니다: 위의 7가지 시나리오 중 가장 고통스러운 하나를 골라 30줄짜리 셸 스크립트를 작성하고, cron에 일주일 등록해 버텨내는지 확인합니다. 버텨내면 두 번째를 추가합니다.

한 번에 7가지를 모두 하려 하지 마세요. 리듬이 맞춰지는 데 시간이 걸립니다.
