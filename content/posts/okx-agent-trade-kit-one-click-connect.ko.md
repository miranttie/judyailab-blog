## OKX Agent Trade Kit 원클릭 연결이란

Agent Trade Kit은 OKX가 AI Agent를 위해 설계한 통합 패키지이며, 원클릭 연결은 새롭게 출시된 인증 메커니즘입니다.

기존 방식은 개발자가 거래소 백오피스에 수동으로 로그인 후 API Key 생성, IP 화이트리스트 및 권한 설정, 키 복사, 설정 파일 입력 순서로 진행되었습니다. 이 과정에는 오류가 발생하기 쉬운 여러 단계가 있으며, 키가 유출될 경우 결과는 심각합니다.

원클릭 연결 방식은 AI Agent가 연결 요청을 시작하면, 사용자가 브라우저에서 OKX 로그인을 완료하고 권한 범위를 선택하는 것으로 끝납니다. 어떤 키 문자열도 수동으로 다룰 필요가 없습니다.

### 세 가지 독립적인 권한 범위

권한은 세 종류로 나뉘며 자유롭게 조합할 수 있습니다.

- **Read(읽기)**: 계정 잔액, 포지션, 거래 내역, 시장 데이터 조회
- **Trade(트레이딩)**: 주문 생성, 취소, 포지션 조정
- **Earn(이자)**: 유동 및 고정 이자 상품 관리

Agent가 시장 분석과 포지션 보고만 수행한다면 Read 권한만으로 충분합니다. 불필요한 권한은 부여하지 않습니다. 최소 권한 원칙이 매우 직관적으로 실행됩니다.

### 7일 자동 만료 메커니즘

사용하지 않는 인증은 7일 후 자동으로 만료됩니다. 이 설계는 중요합니다. 많은 사람들이 테스트 후 기존 인증을 취소하는 것을 잊어버리는데, 원클릭 연결의 자동 만료 메커니즘이 장기 노출 위험을 줄여줍니다.

인증된 모든 Agent는 OKX 백오피스의 "서드파티 인증 관리" 페이지에서 통합하여 관리하고 취소할 수 있으며, API Key 목록을 뒤질 필요가 없습니다.

### 현재 출시된 기능

현재 Agent Trade Kit에 통합된 기능에는 시장 스크리닝, 미결 계약(OI) 분석, 심리 레이더, 스마트 머니 신호, 스마트 이자 관리가 포함됩니다. 이러한 도구들을 조합하면 퀀트 트레이딩 Agent의 정보 입력 요구사항을 기본적으로 충족합니다.

## 이것이 AI Agent 개발자에게 중요한 이유

직접 경험으로 설명하겠습니다.

올해 3월 LabLab.ai의 ERC-8004 해커톤 기간에, API 인증 문제를 처리하는 데만 상당한 시간을 소비했습니다. Demo 환경과 Live 환경의 Key를 따로 관리해야 했고, 서로 다른 거래쌍의 API 엔드포인트 동작에 미묘한 차이가 있었습니다. 게다가 해커톤 규정상 "Agent Trade Kit을 통해 실행된 주문만 리더보드에 반영"되었기 때문에, 전체 주문 흐름을 완전히 재작성해야 했습니다.

4월 13일에는 기존 ccxt 기반 트레이딩 실행 레이어 전체를 OKX Agent Trade Kit CLI로 마이그레이션했습니다. 마이그레이션 자체는 어렵지 않았지만, 원클릭 연결 같은 메커니즘이 처음부터 방지할 수 있었을 함정들이 있었습니다. 명확한 권한 범위, 통합된 설정 관리 진입점, 키 문자열을 수동으로 옮길 필요 없음이 그것입니다.

### 3가지 전략 x 6가지 시장 상태 = 36칸 매트릭스

WaveRider는 세 가지 상호 보완적 전략을 사용했습니다. WaveRider 본체(EMA + RSI + 거래량 확인), BB Squeeze(볼린저 밴드 수축 돌파), MACD 다이버전스. 세 전략은 각각 추세, 돌파, 반전 시장 유형에 대응합니다.

ADX, 볼린저 밴드 너비, EMA 수렴도를 사용해 시장 상태를 감지하여 여섯 가지 유형으로 분류했습니다. 여기에 여섯 개 거래쌍을 결합하여 36칸 전략 매트릭스를 구성했습니다. 각 칸마다 워크포워드 최적화(WFO)를 통해 조정된 파라미터를 적용했으며, 샘플 외 승률은 82.2%를 달성했습니다.

### 7겹 리스크 관리, 최대 낙폭 0.4%

10만 달러 모의 포트폴리오에서 최대 낙폭은 0.4%에 불과했습니다. 이 수치는 전략이 뛰어나서가 아니라, 7겹의 리스크 관리가 엄격하게 실행된 결과입니다.

1. 단일 포지션 한도(최대 5%)
2. 일일 손실 한도(3% 강제 중단)
3. 최대 낙폭 보호(10% 전량 청산)
4. 글로벌 연속 손실 일시 정지 메커니즘
5. 연속 손실 시 포지션 축소(기존의 50%로 감소)
6. 거래쌍 쿨다운 메커니즘(동일 거래쌍 연속 2회 손실 시 3 스캔 주기 쿨다운)
7. 분할 익절(TP1에서 1.5:1 비율로 25% 청산, TP2에서 3.0:1 비율로 25% 청산, 나머지는 추적 손절로 관리)

### 가장 중요한 교훈을 얻은 타임아웃 사건

해커톤 92번째 스캔 시, 중요한 일이 발생했습니다. 통합된 세 AI 공급업체(시장 분석, 신호 확인 등 담당)가 동시에 타임아웃되었습니다.

전체 AI 분석 프로세스가 중단되었습니다.

그때 규칙 엔진이 직접 인수했습니다. 계속 실행하고, 기술 지표를 기반으로 계속 신호를 출력하고, 리스크 관리를 계속 실행했습니다. 충돌도 없었고, 무분별한 주문도 없었고, 중단도 없었습니다.

이 사건은 "AI는 도구이지 목발이 아니다"라는 말을 구호에서 진정한 설계 원칙으로 바꾸어 주었습니다. AI 분석에 의존하는 모든 트레이딩 시스템은 AI에 의존하지 않는 규칙 엔진을 기저 보장으로 갖추어야 합니다.

## 진행 중: OKX OnchainOS 대회

5월 8일, OKX의 Agentic Wallet Trading Competition(OnchainOS)에 등록했습니다. 대회 기간은 5월 8일부터 21일까지이며, 상금 풀은 5만 달러입니다.

이번 대회는 온체인 DEX 트레이딩으로, CEX와는 완전히 다릅니다. onchainos CLI를 MCP server로 사용하며, dex-signal, dex-market, dex-swap, 보안 스캔, 토큰 분석, trenches 밈코인 스크리닝 등 15개 이상의 도구 기능을 개방합니다.

### 5가지 전략 배분

- 스마트 머니 팔로잉(30%): 온체인 실제 고래의 진입 행동 추적
- 거래량 모멘텀 돌파(25%): 대량 거래 이상 신호와 방향 확인 결합
- 다중 시간 프레임 추세(25%): 다중 주기 정렬 후 진입하여 거짓 돌파율 감소
- 크로스 DEX 차익거래(15%): 가격 차이 기회, 실행 속도 요구 가장 높음
- 필터링된 밈코인 저격(5%): 소액 자금으로 고위험 고수익 기회 실행

각 거래 리스크는 2%로 제한하며, 총 자금은 500달러입니다. 이것은 "높은 기준의 리스크 관리 + 소액으로 전략 유효성 검증"하는 설계로, 이전 해커톤의 사고방식을 그대로 이어받은 것입니다.

## 빠른 시작: 5분 만에 OKX Agent Trade Kit 연결하기

직접 시도해 보고 싶다면, 가장 빠른 방법은 다음과 같습니다.

### 1단계: CLI 설치

```bash
npm install -g @okx_ai/okx-trade-cli
```

`okx --version`을 실행하여 설치를 확인합니다.

### 2단계: 계정 설정

프로젝트 디렉토리에 `~/.okx/config.toml`을 생성하고 API 인증 정보를 입력합니다. 먼저 Demo 모드로 테스트하는 것을 권장합니다:

```toml
[default]
api_key = "API Key"
secret_key = "Secret Key"
passphrase = "Passphrase"
```

중요: 모든 설정은 config.toml 한 곳에서만 관리합니다. 환경 변수와 혼용하면 충돌이 발생합니다.

### 3단계: 원클릭 연결 인증

원클릭 연결을 사용하는 경우(수동 API Key 생성 불필요):

1. AI Agent에서 연결 요청을 시작합니다
2. 브라우저에 OKX 로그인 페이지가 나타납니다
3. 로그인 후 권한 범위를 선택합니다 (Read / Trade / Earn)
4. 완료. 비밀 키를 복사하거나 붙여넣을 필요가 없습니다

### 4단계: 연결 테스트

market 모듈로 연결을 확인합니다:

```bash
okx market ticker --instId BTC-USDT-SWAP
```

실시간 가격이 보이면 연결 성공입니다.

### 5단계: 첫 번째 테스트 주문

Demo 모드에서 소액 시장가 주문을 실행합니다:

```bash
okx swap place --instId BTC-USDT-SWAP --side buy --ordType market --sz 1
```

주문이 확인되면 Agent Trade Kit 연결이 완료된 것입니다. 이제 이 명령어들을 AI Agent의 실행 루프에 통합하면 됩니다.

## Agent Trade Kit 통합 실제 함정 기록

ccxt에서 Agent Trade Kit으로의 마이그레이션 과정은 순조로웠지만, 다시 같은 함정에 빠지지 않도록 특별히 주의할 몇 가지 사항이 있습니다.

### 문제 1: Demo와 Live 모드 설정 충돌

Demo 모드와 Live 모드의 연결 엔드포인트가 다르기 때문에, 환경 변수와 config.toml을 동시에 사용하면 "환경 변수를 분명히 변경했는데 여전히 기존 엔드포인트에 연결된다"는 문제가 쉽게 발생합니다.

해결책은 **config.toml 하나의 소스로만 모든 설정을 관리하고** 환경 변수와 혼용하지 않는 것입니다. 환경을 전환할 때마다 해당 config.toml 파일로 교체하고, 두 곳에 동시에 설정값을 두지 않습니다.

### 문제 2: CLI 표준 출력이 업데이트 알림으로 오염

Agent Trade Kit CLI는 일부 버전에서 명령 실행 시 "A new version is available"과 같은 업데이트 알림을 함께 출력합니다. 이 텍스트가 JSON 반환값에 섞이면 JSON 파서가 직접 오류를 발생시킵니다.

해결책은 **응답 파싱 전에 비-JSON 행을 먼저 필터링**하는 것입니다. `{` 또는 `[`으로 시작하는 첫 번째 행을 찾아 거기서부터 파싱을 시작하고, 앞의 모든 출력은 무시합니다.

### 문제 3: 서킷 브레이커 보호 부재

CLI 연결 실패 시 서킷 브레이커 메커니즘이 없으면 Agent가 계속 재시도하여, 자원 낭비와 함께 시장이 불안정할 때 중복 주문이나 주문 상태 불일치를 유발할 수 있습니다.

자체 서킷 브레이커를 추가했습니다. **연속 5회 이상 실패 시 5분 쿨다운을 트리거**합니다. 쿨다운 기간에는 새 요청을 전송하지 않으며, 쿨다운 종료 후 연결을 재초기화합니다. 이 메커니즘은 실제 사용 중 문제가 악화되기 전에 여러 차례 이상을 차단했습니다.

### 보충: 일부 거래쌍은 계약 크기 보정이 필요

DOGE, BNB 등 거래쌍의 계약 크기(contract size)는 다른 주요 코인과 다릅니다. 주문 수량을 직접 계산하면 포지션 크기에 편차가 생깁니다. 시스템 시작 시 거래소의 계약 사양을 읽어 계약 크기 보정을 주문 로직에 내장하고, 하드코딩하지 않도록 권장합니다.

전체 Agent 트레이딩 개발 여정을 돌아보면, 인증과 인가 관리는 언제나 "보이지 않는 시간 블랙홀"이었습니다. 가장 매력적인 기술 문제는 아니지만, 실수가 발생할 때마다 전체 개발 리듬을 방해했습니다.

하지만 기술 도구는 기초일 뿐입니다. 두 번의 해커톤에서 얻은 가장 중요한 교훈은 어떤 SDK가 가장 사용하기 편한가가 아니라, 다음과 같습니다.

**AI 분석이 전략을 더 스마트하게 만들지만, 시스템을 안정적으로 운영하는 것은 언제나 예측 가능한 규칙 엔진과 엄격한 리스크 관리 메커니즘입니다.**

원클릭 연결은 "어떻게 연결하는가"의 문제를 해결했습니다. "연결 후 어떻게 할 것인가"는 여전히 탄탄한 엔지니어링 실천으로 답해야 합니다.
